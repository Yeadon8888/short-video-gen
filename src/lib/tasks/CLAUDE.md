# tasks

## 目录结构

- `batch-queue.ts`：批量带货提交节流规则，统一控制每轮商品与 slot 的提交窗口，以及两次 provider 提交之间的间隔。
- `downloads.ts`：把任务结果打包为 ZIP，统一服务端下载入口。
- `expiry.ts`：视频过期清理，负责让 3 天过期策略真正落地。
- `expiry-meta.ts`：过期倒计时的纯计算逻辑，前端展示与后端清理共用一套规则。
- `runner.ts`：统一维护入口，串联视频任务、批量任务与超时处理。
- `result-assets.ts`：把成功视频镜像到自有存储，给 ZIP 下载与过期物理删除提供可控 key。
- `timeout.ts`：超时任务轮询与退款，不处理媒体过期。
- `reconciliation.ts`：标准任务的终态结算与退款。
- `fulfillment.ts`：按 slot 维度做目标补齐、重试与结算。

## 架构决策

任务生命周期拆成四条线：
结算、下载、清理、异步推进。
它们可以共享数据库，但职责绝不混写，否则状态与账务会互相污染。
图片编辑虽然也是异步任务，但属于另一类长耗时媒体处理，不能继续塞进视频维护器的分钟级轮询入口。
批量带货的“创建请求”与“状态轮询”不能混成一个爆发式入口。
提交窗口必须单独建规则层，否则用户一旦批量选多图，provider 限流会直接把整组任务打穿。
只限制每轮数量还不够；provider create 之间必须留出节奏间隔，避免 2 条请求在同一秒内同时命中账号频率限制。

## 开发规范

- ZIP 下载逻辑放服务端，不在前端循环点链接。
- 过期清理优先让应用侧访问失效；远程物理删除需确认底层存储归属后再做。
- 若要做物理删除，任务结果必须先落到自有存储并持久化 key；只有 URL 没有 key 的设计无法安全删除。
- 退款逻辑不要塞进下载或过期清理模块。
- `runner.ts` 只处理视频域维护；跨媒体类型的长任务应拆独立 worker。
- 批量带货提交必须经过窗口节流；不要在 route 或处理器里直接一口气提交整组商品。
- provider create 请求之间保留固定间隔；默认按 2 秒节奏依次提交。
- provider 适配层内部若一次要创建多条视频任务，也必须复用同一节流间隔；不能绕过队列规则自己连发。
