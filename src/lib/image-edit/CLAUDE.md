# image-edit

## 目录结构

- `bltcy.ts`：图片编辑服务适配层，负责挑选图片编辑模型、组请求、解析返回结果。
- `queue.ts`：图片任务队列规则，负责统一定义并发上限与空闲槽位计算。
- `jobs.ts`：图片转换任务仓储与处理器，负责批量入队、状态查询、异步执行与扣费入库。
- `worker.ts`：图片任务内部 worker 触发与鉴权边界，负责把长耗时执行从通用维护器拆出。

## 架构决策

图片编辑不是视频生成的“特例参数”，而是独立能力。
这里单独建目录，是为了把供应商协议、提示词、返回解析，以及异步任务处理边界，全部隔离在图片域内，避免污染主视频链路。
图片编辑 worker 必须独立于视频维护器运行，否则长耗时外部模型调用会拖垮分钟级 tick，造成队列饥饿与假死。
并发策略必须集中在单一位置管理，避免路由、worker、任务处理器各自维护一份并发判断，最终把队列错误地串行化。

## 开发规范

- 路由层不要直接拼 BLTCY 请求体。
- 模型选择必须走 `image_edit` capability。
- 长耗时图片编辑必须走异步任务，不要重新回到同步 HTTP 阻塞链路。
- 图片提交路由只负责入队和触发 worker，不承担实际图片编辑执行。
- 并发上限与槽位计算统一从 `queue.ts` 读取，不要在路由或 UI 里硬编码另一套数字。
- 如果后续增加其他图像能力，优先在本目录新增适配器，不要把逻辑塞回 `video/`。
